# 1. きっかけ

- メモリースポーツに興味がわいてから、数字を単語に変換する辞書を作り始めた
- 大半の人が作るのは2桁(00〜99)までみたいだが、自分は 000〜999 の3桁まで作ることにした
  - 人生の中では3桁の数字に関わることがたくさんあるし、この程度の対応表くらい一度作って持っていてもいいと思った
  - **tips: ベンフォードの法則,ハッシュ**
- イメージとしてはこんな感じ: `124 → 「いにし」`, `893 → 「やくさ」`, `357 → 「ソナー」`

- それでも続いたのは、辞書を埋めていくのがとんでもなく楽しかったから
- spreadsheet をひたすら埋めた

- 電車や移動中もすぐに確認できるように対応表確認ツールを作った
- 練習のためにランダムに数字出すツールも作った

# 4. 辞書ができて得たもの

- IDやPINコードが物に見える (`4729 → 「しなふく」`、`0868 → 「おはろは」` など、数字の並びが勝手にイメージに化ける)
- 円周率100桁を30分で覚えられる
  - 例: `3.14` → `さいし`、`159` → `いこく`、`265` → `ふろこ` のように 3 桁ずつ単語化してストーリー化する
- 1000通りのランダムな
  - 占い・無作為なインスピレーションできる
    - ナンバープレートなど日常で見かけた数字に対してものを連想する
    - 物(辞書になくていい) →数字 → 数字+1 → 物2 もできる
      - 「そうめん」→35/7/69/0 → 「357 ソナー」, 「358 そば」

# 5. 途中で見えてきた面白さ：数字キャラ

## 5-1. 数字キャラという当たり枠に愛着が湧く

- 名前を3桁の数字に変換できるキャラ（数字キャラ）だけを愛でた
- 見つけると稲妻が走る
- 検索で出てきた数字キャラが出るアニメをあえて見る
- 名前が覚えられないキャラは自宅のデスクトップ背景にするなどして覚えた
- ソシャゲのキャラをひたすら追うためのツールも作った
- 愛称があると助かる

## 5-2. 数字キャラ

- （ま=00, お=0）から始まるキャラは、その時点でちょっと残念 (変換表に追加できないから)
- ゲーム内で名前をつけられるときは、余ってる語呂でつける
- なかなか埋まらない 128 は、ゲーム内の相棒に「いにば」とつけたりした

## 5-3. 名簿が豊富な作品

- この観点では、2文字3文字名前が多い「トリッカル」と「ブルーアーカイブ」は神名簿で助けられた。

# 6. 変換表のルールを固めるまで

- 数字変換表のルールは、作りながら何度も入れ替えたり変更したりした
- オリジナルキー配列を作っている人も、こういう試行錯誤をしていそうだと思った

## 6-1. 完成と実用は別

- 辞書が完成してから、変換がすぐにできるかは別の話

## 6-2. 4桁辞書への未遂

- 4桁辞書は準備はしたけど、3桁で手一杯だなと思った
  - 「数字toかな」は一意でないため、桁が増えるほど想起が困難になる
    - 例: `1234` は `[1][2][34] = い+ふ+みょ` とも `[1][23][4] = い+つ+し` とも分解でき、組合せ爆発する
  - 変換表で埋まらない部分が多くて心が折れた

---

# 7. テンキー入力や「かな表」の試行錯誤

- 2桁=かな表を作っても、かな2文字を割り当てるのは難しい
- 3桁の場合、すべて4数字目を切り捨てることになる
- 定着する前に毎回五択を迫られるのはどうなのか、検討しきれなかった
- 五十音を10x10に割り当てきれない
- 3文字以上の言葉を先頭2文字から連想するのは、想像以上に難しい
- 自分の辞書はそうしていない
- ただし、反復で容易になるものでもあり、みんなやっているのかもしれない
- とはいえ、余ってる平仮名の一部はこのルールで使っている
  - 33=す
  - 73=ぬ
  - 35=そ
  - 75=の

# 8. ルールの変遷

ここまでが「やってみたが分かったこと」ベースの話。ここからは、変換ルール本体がどう進化していったかを版数で振り返る。

## 8-1. v1.1 — 「ん」の導入

初期ルールでは、「ん」が存在しなかった。

これは一見そこまで問題に見えないが、実際に単語生成を始めるとかなり苦しい。例えば日本語では、

- 〜ん
- ん〜
- ○ん○

が極めて頻出する。しかし「ん」が無いと、

- 単語が急に不自然になる
- 語尾が制限される
- 人名生成力が落ちる

などの問題が出た。そこで `0` に「ん」を割り当てた。

これはかなり革命的で、

- 日本語らしい音列が急増
- 語尾処理が楽になる
- 圧縮率が上がる

という効果があった。一方で、

`0 = れ / お / ま / ん`

みたいな状態になり、さらに候補爆発が悪化していく。

## 8-2. v2.0 — 拗音ルールの誕生

ここで初めて、「単語を後付け生成する」のではなく、

> 日本語の音節構造そのものを圧縮する

方向へ進み始める。

### 初代拗音ルール

当時のルールはかなり単純だった。

- `X0 → ょ`
- `X8 → ゃ`
- `X4 → ゅ`

例えば:

- 20 → ちょ
- 68 → りゃ
- 64 → りゅ

のような感じ。

ただし、このルールは「イ段を持つ文字」のみ適用された。つまり、

- き
- し
- ち
- に
- ひ
- り

など。これはかなり重要な転換点だった。それまで:

> 数字 → 文字

だったものが、ここで初めて:

> 数字列 → 音節

へ進化した。

### やゆよ数字の変遷

実はこの割り当ては長期間安定しなかった。特に、

- ゃ
- ゅ
- ょ

の位置は何度も変更された。最終的には、

- 8 → ゃ
- 7 → ゆ
- 4 → よ

に近い構造へ収束していく。このあたりは、

> 「作りやすさ」と「復号しやすさ」

のせめぎ合いだった。拗音は便利だが、自由にしすぎると逆に読めなくなる。

## 8-3. v2.1 — 本格2桁時代

ここでルールが一気に体系化される。

### 母音割り当て

例えば:

- 35 → そ
- 65 → も
- 75 → の

のように、後半数字が母音を決める構造が導入される。この時期から、

- u
- e
- o

など、母音ベースで整理し始めた。つまり、

> 「数字を単語に変換する」

というより、

> 「数字を音素へ変換する」

設計に近づいていく。

### 特殊2桁

頻出音は例外として独立した。例えば:

- 55 → た
- 22 → ち
- 33 → す
- 39 → せ

など。ここはかなり人力最適化感が強い。実際には、

- 頻出
- 作りやすい
- 読みやすい
- 一瞬で復号できる

などの経験則で決まっていった。

### 1桁テーブル削減

この頃になると、初期の「何でもあり1桁テーブル」は限界を迎える。候補が多すぎると、

- 読めない
- 固定できない
- 記憶できない

ことが分かってきた。結果として、各数字の1桁候補は最大3つ程度まで圧縮された。これはかなり大きな思想転換で、

> 自由度を下げることで、逆に運用性を上げる

方向へ進み始めた時代だった。

# 9. ルール本体: スコアリングの仕組み

ここまでの試行錯誤を経て、最終盤にたどり着いたのがこのスコアリングルール。トークン (= かな1〜2文字) ごとにスコアを足し引きして単語の良し悪しを定量化する。実装は [`src/visualize-scoring.html`](https://github.com/elzup/999/blob/main/src/visualize-scoring.html) でビジュアライズしている。

ざっくりイメージ:

| 単語       | 分解              |            スコア | コメント                          |
| ---------- | ----------------- | ----------------: | --------------------------------- |
| `ちき`     | `[22][9]`         |  30 + 10 = **40** | 2 桁 + 1 桁の最強パターン         |
| `いす`     | `[1][33]`         |  10 + 30 = **40** | 1 桁 + 2 桁、こちらも 40 点       |
| `いにさ`   | `[1][2][3]`       | 10 + 10 + 10 = 30 | 1 桁 ×3、無難                     |
| `しょうき` | `[44][省略う][9]` |  30 - 5 + 10 = 35 | 拗音 + 「う」省略のペナで微減     |
| `ちす`     | `[22][33]` (4 字) |       30 + 4 = 34 | 4 字目が境界をまたぐ halfOverflow |

同じ 3 桁でも分解の仕方でスコアが変わる。辞書を埋めるときは「短くて高得点」な候補を優先する。

## スコア重み

| トークン種別       |    重み | 概要                                         |
| ------------------ | ------: | -------------------------------------------- |
| `core` (1 桁 core) | **+10** | 各桁の最優先かな (例: 1=い, 2=に)            |
| `sub` (1 桁 sub)   |  **+8** | 同じ桁の代替かな (例: 1=ひ, 2=ふ)            |
| `bad` (1 桁 bad)   |  **+6** | 苦肉のかな (例: 0=れ, 9=か)                  |
| `double` (2 桁)    | **+30** | 1〜2 文字で 2 桁を表せる強いトークン         |
| `sokuon` (促音 っ) | **+20** | 次のかなの数字を repeat                      |
| `halfOverflow`     |  **+4** | 2 桁トークンが境界をまたぐペナルティ込み残点 |
| `overflowPerChar`  | **-10** | 3 桁を完全に越えた文字                       |
| `youon4Omission`   |  **-5** | 拗音の中で「う/ん」を省略する場合のペナ      |

| **sub (+8)**   | お  | ひ  | ふ  | み  | よ  | ら  | る  | う  | や  | く  |

## トークン分類フロー

入力単語を頭から舐めて、各かなトークンを以下の判定で分類する。

```
Token 入力
  ↓
startPos >= 3 ?     ─ Yes →  overflow (-10)
  ↓ No
っ/ッ ?              ─ Yes →  sokuon (+20)
  ↓ No
2 桁 && endPos >= 3 ?─ Yes →  halfOverflow (+4)
  ↓ No
2 桁 ?               ─ Yes →  double (+30)
  ↓ No
single (tier 別)          →  core +10 / sub +8 / bad +6
```

## 位置 × トークン種別スコア

3 桁ターゲット時、各トークン種別が各位置で得るスコア。

| Token Type    |     pos 0 |     pos 1 |     pos 2 | pos 2-3 (境界) | pos 3+ (overflow) |
| ------------- | --------: | --------: | --------: | -------------: | ----------------: |
| single (core) |       +10 |       +10 |       +10 |              — |               -10 |
| single (sub)  |        +8 |        +8 |        +8 |              — |               -10 |
| single (bad)  |        +6 |        +6 |        +6 |              — |               -10 |
| double (2 桁) | +30 (0-1) | +30 (1-2) | +30 (1-2) |       +4 (2-3) |               -10 |
| sokuon (っ)   |       +20 |       +20 |       +20 |              — |               -10 |

## 3 桁を作る分解パターン比較

| パターン                      |       スコア | 概要                                             | 例                                        |
| ----------------------------- | -----------: | ------------------------------------------------ | ----------------------------------------- |
| `[DD][D]` double + single     | **36〜40** ★ | 2 桁(30) + 1 桁(6〜10)                           | `ちき → [22][9] = 40`                     |
| `[D][DD]` single + double     | **36〜40** ★ | 1 桁(6〜10) + 2 桁(30)                           | `いす → [1][33] = 40`                     |
| `[D][D][D]` all singles       |       18〜30 | 1 桁 ×3                                          | `いにさ → [1][2][3] = 30`                 |
| `[っ][D][D]` sokuon + singles |       32〜40 | 促音(20) + 1 桁 ×2 ※次の数字 repeat              | `っきき → [っ9][9] sokuon=20 + single=10` |
| `[DD][DD]` double overflow    |           34 | 2 桁 + 半越え 2 桁 (4 字 → 3 桁)                 | `ちす → [22][33] = 30+4 = 34`             |
| `youon4 省略`                 |           35 | 拗音 2 桁(30) + 省略(-5) + 1 桁(10) ※4 字 → 3 桁 | `しょうき → [44][省略う][9] = 35`         |

最強はやはり「2 桁トークン + 1 桁」の組み合わせ。短い単語で 40 点が取れる組み合わせを優先的に埋めるのが効率良かった。

  - 実用性ありそう
